# Starbucks Review Analyzer

## General Information

- **Project Title:** Starbucks Customer Review Sentiment Analysis Application
- **Description:** An AI-powered web application that analyzes Starbucks customer reviews through three deep learning pipelines: **Sentiment Analysis**, **Text Summarization**, and **Automated Service Reply Generation**, enabling data-driven customer experience management.

Built as part of the ISOM5240 Group Project.

- **GitHub Repository:** https://github.com/evelynlyn1489-cell/Starbucks-Review-Analyzer

## Project Description

Starbucks receives thousands of customer reviews across online platforms. Manually reading and responding to each review is time-consuming and inconsistent. This application automates the process using three Hugging Face pipelines:

1. **Sentiment Analysis** classifies each review as Positive or Negative with a confidence score. Reviews with low confidence (< 75%) are flagged as ambiguous for manual review.
2. **Text Summarization** condenses lengthy reviews into 1-2 key sentences, allowing managers to quickly understand customer concerns without reading the full text.
3. **Automated Service Reply** drafts a personalized response — apologetic for negative reviews, grateful for positive ones — that staff can edit before sending.

## Methodological Information

### Data Collection

- **Source:** Kaggle — Starbucks customer reviews from Consumer Affairs
- **Collection Method:** Web-scraped reviews with star ratings (1-5), reviewer name, location, date, review text, and image links
- **Time Period:** Reviews spanning 2000-2023, concentrated in 2014-2017

### Data Processing

1. Removed 37 entries with "No Review Text" (placeholder entries with no actual content)
2. Removed 110 entries with missing star ratings (no ground truth for labeling)
3. Removed duplicate rows
4. Mapped star ratings to binary sentiment labels: 1-3 stars as Negative (label 0), 4-5 stars as Positive (label 1)
5. Applied light text cleaning: removed URLs, email addresses, normalized whitespace, replaced `**` name masks with `[NAME]`
6. Split data 70/15/15 into train/validation/test sets using stratified sampling
7. Applied random oversampling on the minority class (Positive) in the training set only to achieve class balance

### Model Selection Process

Each pipeline evaluated 3 candidate models. Selection criteria were accuracy (for sentiment) or output quality (for summary/reply), runtime speed, and suitability for deployment on Streamlit Cloud (CPU-only, limited memory).

**Pipeline 1 — Sentiment Analysis:**
- distilbert-base-uncased-finetuned-sst-2-english (selected — highest accuracy 95.05%)
- cardiffnlp/twitter-roberta-base-sentiment-latest (93.07%)
- nlptown/bert-base-multilingual-uncased-sentiment (90.10%)

**Pipeline 2 — Summarization:**
- MBZUAI/LaMini-Flan-T5-248M (selected — best quality-to-speed ratio)
- google/flan-t5-small
- facebook/bart-large-cnn

**Pipeline 3 — Auto Reply:**
- MBZUAI/LaMini-Flan-T5-248M (selected — best reply quality for deployment)
- google/flan-t5-small
- google/flan-t5-base

- All code tested with "Restart & Run All" in Jupyter to ensure reproducibility
- Stratified train/test splitting preserves label distribution across sets

- Model evaluation uses held-out test set never seen during training
